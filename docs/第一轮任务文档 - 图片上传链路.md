# 第一轮任务文档 - 图片上传链路

## 1. 文档目标

本轮任务目标是打通图片上传链路的最小闭环：

```text
上传图片
→ 校验通过
→ 保存到本地文件系统
→ 写入 image_asset 表
→ image_status = NORMAL
→ vector_status = PENDING
→ 返回 imageId / imageUrl / 状态
```

本轮不处理完整图片向量化执行流程，不实现搜索、分页查询、详情查询、删除等非上传主链路能力。

---

## 2. 上传链路组件类

组件类只负责单一能力，不负责完整流程编排。

### 2.1 上传参数与基础校验组件

| 类 | 任务说明 |
|---|---|
| `DefaultImageUploadValidator` | 实现上传文件基础校验，包括文件非空、文件大小、扩展名、MIME 类型、图片是否可解析 |
| `StorageProperties` | 绑定本地存储配置，包括 `basePath`、`accessUrlPrefix`、`maxFileSize`、`allowedExtensions` |

说明：

`DefaultImageUploadValidator` 只判断上传文件是否合法，不负责保存文件、不负责入库、不负责返回响应。

---

### 2.2 图片信息解析组件

| 类 | 任务说明 |
|---|---|
| `DefaultImageMetadataExtractor` | 解析图片基础信息，包括 `fileExt`、`mimeType`、`width`、`height`、`fileSize` |
| `ImageMetadata` | 承载图片基础元数据解析结果 |

说明：

该组件只负责读取图片元数据，不负责去重、不负责文件存储、不负责数据库入库。

---

### 2.3 文件 Hash 与去重组件

| 类 | 任务说明 |
|---|---|
| `Sha256FileHashCalculator` | 基于文件内容计算 SHA-256 hash |
| `DefaultImageDuplicateChecker` | 根据 `fileHash` 判断图片是否重复，重复时抛出业务异常 |
| `ImageAssetRepository.existsByFileHash` | 提供按 `fileHash` 查询是否存在图片的能力 |

说明：

去重只基于文件内容 hash，不基于原始文件名。重复图片直接拒绝，不保存文件、不写入 MySQL、不发布向量化任务。

---

### 2.4 ID 与文件名生成组件

| 类 | 任务说明 |
|---|---|
| `SnowflakeImageIdGenerator` | 生成图片主键 ID |
| `DefaultImageFileNameGenerator` | 根据 `imageId + fileExt` 生成系统文件名，例如 `100001.jpg` |

说明：

`imageId` 既是 MySQL 图片主键，也会作为后续向量库中的 `vectorId` 使用，因此需要保证全局唯一。

---

### 2.5 文件存储组件

| 类 | 任务说明 |
|---|---|
| `DateBasedStoragePathGenerator` | 根据日期、图片 ID、扩展名生成存储路径，例如 `/images/yyyy/MM/dd/imageId.ext` |
| `LocalFileStorageService` | 实现本地文件保存、删除、读取、存在性判断、访问 URL 生成 |
| `StorageSaveRequest` | 文件保存请求对象 |
| `StoredFile` | 文件保存成功后的返回对象 |
| `StoragePathContext` | 路径生成上下文 |
| `StorageResource` | 文件读取结果对象，主要供后续向量化流程使用 |

说明：

本轮上传链路优先实现：

```text
save
delete
getAccessUrl
```

`read` 和 `exists` 可先保留 TODO，后续图片向量化链路再完善。

---

### 2.6 MySQL 入库组件

| 类 | 任务说明 |
|---|---|
| `ImageAssetMapper` | 新增 MyBatis-Plus Mapper，用于直接操作 `image_asset` 表 |
| `ImageAssetPO` | 对应 `image_asset` 表字段 |
| `ImageAssetSaveCommand` | 应用层传给 Repository 的入库命令 |
| `ImageAssetRepositoryImpl.save` | 将 `ImageAssetSaveCommand` 转换为 `ImageAssetPO` 并保存到 MySQL |
| `ImageAssetRepositoryImpl.existsByFileHash` | 根据 `fileHash` 查询是否存在重复图片 |

说明：

这是本轮必须补齐的底层能力。没有 `ImageAssetMapper`，Repository 无法真正落库。

---

### 2.7 状态与错误组件

| 类 | 任务说明 |
|---|---|
| `ImageStatus` | 定义图片状态：`NORMAL`、`DELETED`、`INVALID` |
| `VectorStatus` | 定义向量状态：`PENDING`、`PROCESSING`、`READY`、`FAILED` |
| `ImageAssetErrorCode` | 定义图片上传链路错误码 |
| `BizException` | 上传校验失败、重复图片等业务错误统一抛出 |
| `GlobalExceptionHandler` | 将业务异常和系统异常统一转换为 `Result.fail` |

说明：

上传成功时需要初始化：

```text
imageStatus = NORMAL
vectorStatus = PENDING
```

---

### 2.8 日志、指标、补偿组件

| 类 | 任务说明 |
|---|---|
| `Slf4jLogService` | 记录上传链路开始、成功、失败、异常、状态变化等日志 |
| `LogMetricRecorder` | 记录上传次数、上传耗时、失败次数等指标 |
| `LogOrphanFileLogService` | 文件保存成功但 MySQL 入库失败，且补偿删除失败时记录孤儿文件 |
| `OrphanFileRecord` | 孤儿文件记录对象 |

说明：

这些类不影响上传最小闭环，可以先做简单实现。`LogOrphanFileLogService` 需要在 MySQL 入库失败且文件删除失败时使用。

---

### 2.9 请求上下文组件

| 类 | 任务说明 |
|---|---|
| `RequestContextFilter` | 在 HTTP 请求进入时初始化 `requestId` 和 `traceId` |
| `DefaultTraceIdGenerator` | 生成 `requestId` 和 `traceId` |
| `RequestContextHolder` | 基于 ThreadLocal 保存当前请求上下文 |

说明：

这部分不直接影响文件保存和入库，但会影响统一响应中的 `requestId`，建议在上传链路联通后补齐。

---

## 3. 上传链路业务编排类

业务编排类负责调用多个组件，并决定流程顺序、异常分支和补偿策略。

### 3.1 接口层编排

| 类 | 任务说明 |
|---|---|
| `ImageAssetController` | 接收 multipart 上传请求，构造 `ImageUploadCommand`，调用 `ImageAssetApplicationService.upload`，将 `ImageUploadResult` 转换为 `ImageUploadResponse` 并返回 `Result` |

说明：

Controller 不做复杂业务逻辑，不直接调用存储、数据库、去重、向量化任务发布等组件。

---

### 3.2 应用层核心编排

| 类 | 任务说明 |
|---|---|
| `DefaultImageAssetApplicationService` | 实现完整上传主流程，编排校验、元数据解析、hash 计算、去重、ID 生成、文件保存、MySQL 入库、向量化任务发布、异常补偿和结果返回 |

说明：

这是本轮最核心的业务类。上传链路的主流程、异常分支、补偿逻辑都集中在这里。

核心流程：

```text
validate
→ extract metadata
→ calculate hash
→ check duplicate
→ generate imageId
→ generate fileName
→ save file
→ save image_asset
→ publish vectorization task
→ build upload result
```

异常分支：

```text
文件保存失败
→ 不写 MySQL
→ 返回上传失败

MySQL 入库失败
→ 删除已保存文件
→ 删除失败则记录孤儿文件

向量化任务发布失败
→ 不影响上传成功
→ 保留 vectorStatus = PENDING
→ 记录失败日志
```

---

### 3.3 持久化编排

| 类 | 任务说明 |
|---|---|
| `ImageAssetRepositoryImpl` | 编排 `ImageAssetSaveCommand` 到 `ImageAssetPO` 的转换，并调用 `ImageAssetMapper` 完成数据库写入和查询 |

说明：

Repository 不负责上传主流程，只负责图片资产数据的持久化适配。

---

### 3.4 任务发布编排

| 类 | 任务说明 |
|---|---|
| `ThreadPoolVectorizationTaskPublisher` | 接收 `ImageVectorizationTaskCommand`，在 MVP 阶段将任务提交到本地线程池，或先返回发布成功占位结果 |

说明：

本轮可以先做最小实现：只返回发布成功或记录日志，不真正执行向量化。真正图片向量化链路放到第二轮任务。
